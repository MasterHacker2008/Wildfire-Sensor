Wildfire Detector

This project uses an embedded system to collect real-time environmental data and a computer-based model to estimate wildfire risk.

Meeting the brief

Basic Requirements

I developed a functional embedded system that collects environmental data related to wildfire risk. It uses analogue inputs (temperature and humidity) and a digital input (a button). The system generates a digital output through an LED when risk is high it turns red. (an adaptive feedback mechanism) and automatically sends data to a server. After initial setup, it runs independently, continuously monitoring and calculating fire risk.

The collected data is stored and used in my computer-based model. The system simulates how high temperature, low humidity increase wildfire risk, and I tested it under different environmental conditions.

Advanced Requirements

Using Python, I developed a wildfire risk model (logistic regression model) that is trained on an external dataset and is tested on sensor data collected from the embedded system to predict risk. I also tested “what-if” scenarios. (In the simulator section)

Investigation

After brainstorming several ideas such as drought monitoring, canopy cover, and species population tracking, I decided to focus on wildfire risk monitoring because it was the simplest one to implement.

Wildfires are a major environmental issue that can destroy forests, reduce biodiversity, and release large amounts of carbon dioxide. Monitoring environmental conditions linked to wildfire risk is therefore highly relevant in a real-world scenario.

Another reason for choosing this idea is that it can be built using simple sensors such as temperature, humidity, and soil moisture sensors (Sensors that are inexpensive and that I have access to), which are directly linked to wildfire risk. High temperatures, low humidity, and dry soil conditions increase the likelihood of fires starting.

This project also aligns well with the brief. It allows the development of a computer-based model that analyses collected data and calculates wildfire risk levels. The model can also test “what-if” scenarios, such as how risk changes if temperature or humidity changes.

I then researched existing wildfire monitoring systems. One example is a system developed by Dryad Networks, which uses solar-powered gas sensors placed throughout forests to detect gases produced by fires and alert responders through an IoT network.

Another solution by Pyri uses heat-activated sensors distributed across forests. When high temperatures are detected, the sensors send alerts through mobile networks.

These systems influenced my project by demonstrating the importance of sensor networks for early wildfire monitoring. Inspired by this, my project focuses on collecting environmental data such as temperature and humidity to build a predictive model that estimates wildfire risk and provides early warnings when conditions become dangerous.

My Goal

The goal of my project is to develop a system that collects environmental data such as temperature and humidity and uses this data to predict wildfire risk even before fires occur.

Plan & Design

Project Options and Design Choices

The original plan was to use a Micro:bit as the microcontroller. However, transmitting sensor values (particularly the button as it needs to send an alert) to the website required using USB serial communication. During testing, I discovered that serial communication would occasionally split and merge data packets, making transmission unreliable.

The image here shows the serial output where the temperature 23 sometimes appears as just as 3. Even with using seperators (/) this issue remains, making it impossible for accurate predictions to be made.

To solve this, I considered sending data directly to the website over WiFi. However, the Micro:bit did not work well with the ESP serial WiFi module available to me. Due to these limitations, I decided to switch to a Raspberry Pi Pico which works with the esp. Since I had prior experience working with the pico (I used it for my JC tech project) this was not a setback.

Research into Different Models

I explored several approaches for predicting wildfire risk:

  • ML Models:
    • Logistic Regression - a standard statistical model.
    • Random Forest - a more complex machine learning model.
  • Simple Mathematical Model
Choosing a Model

I decided against a fully algorithmic approach because determining the influence of each environmental factor on wildfire risk would require a lot of research and complex calculations. Instead, I chose a model that could be trained using an existing dataset.

Since my knowledge of machine learning is limited (I initially thought the Random Forest model had something to do with forests), I avoided complex models and selected a simple but effective one: Logistic Regression.

This allowed me to focus on building the server and programming the Pico, which are the areas I am strongest in.

I found a California wildfire dataset (from kaggle.com) containing environmental factors and a target variable indicating whether a wildfire occurred. I used this dataset to train the logistic regression model, allowing it to predict wildfire risk based on sensor data collected from the embedded system.

To simplify the model, I calculated feature importance using Python to identify the factors that strongly influence wildfire risk. Ideally, the model would include temperature, precipitation, and solar radiation. However, sensors for precipitation and solar radiation were not available, so I selected temperature and humidity as the key variables for training the model.

Stakeholders & User Needs

The main stakeholders for this system include forest authorities, environmental agencies, local communities, and wildfire researchers. They require reliable monitoring to help detect wildfire risks early. Key user needs include clear risk levels (Low-Extreme), early warning alerts, and automated monitoring. To meet these needs, the system continuously collects environmental data, calculates wildfire risk using a machine learning model and displays live data on a web dashboard.

Create

Version 1 - Machine Learning Model

Sprint 1 - Training the Prediction Model

In the first stage of my project I focused on creating the Logistic Regression model.
Logistic regression is a ML algorithm used for binary classification, meaning it predicts either wildfire risk (1) or no risk (0). The model was trained using an external California wildfire dataset with temperature and humidity as input features and the target column (0 or 1) indicating whether a wildfire occurred. After training, the model uses temperature and humidity from the sensors as inputs to calculate a probability between 0 and 1 representing the wildfire risk.

The trained model was exported as a .joblib file so it could be used later for predictions. Initial testing showed an accuracy of 95.7%, which I thought was promising.

Sprint 2 - Attempting Microcontroller Deployment

The next goal was to run the trained model directly on the Pico so that predictions could be made locally.

However, this was not possible due to the limitations of the microcontroller. Devices such as the Pico have limited RAM and cannot support large libraries such as scikit-learn. Therefore, the model could not run on the Pico.

This meant I had to redesign the system architecture.

Version 2 - Backend Server

Sprint 3 - Architecture Redesign

To overcome this problem, the ML model was moved to a server instead. In this new architecture, the Raspberry Pi Pico sends sensor data to the server, which runs the model and returns a wildfire risk prediction.

Because the model was built in Python, a Python backend framework was required. I had mainly worked with Node.js (A javascript framework to make servers) before, but since the machine learning model was built in Python, using a Python framework made more sense. After some research, I chose FastAPI. Although I was new to it, the clear documentation allowed me to build the server by just copy pasting the necessary code with a few tweaks.

Sprint 4 - Backend Development

The FastAPI server became the main component of the system. It receives sensor data, runs the machine learning model, and returns wildfire risk.

To accomplish this, several API endpoints were created:

  • POST /data - Receives sensor readings and stores the data
  • POST /predict - Generates predictions
  • GET /data - Retrieves stored historical records
  • WebSocket /ws - Sends real-time updates to connected clients (the website)

Instead of using a traditional database (because I found it difficult), data was stored in Google Sheets using the Sheety API, which provided a simple cloud storage solution that was easy to inspect and edit.

Version 3 - Embedded System

Sprint 5 - Raspberry Pi Pico Integration

The hardware system was developed using the Pico, a DHT11 temperature and humidity sensor, an ESP-01 WiFi module, an RGB LED, and a push button.

The DHT11 sensor periodically measures temperature and humidity, which are then sent to the server for prediction. A button press triggers a m_alert = true (manual alert), which displays a popup in the website alerting the user immediately.

Sprint 6 - RGB LED Risk Indicator

An RGB LED was implemented to provide a visual indicator of wildfire risk. Pins GP16, GP17, and GP18 were used to control the LED colours Red, Green, and Blue.

A function called riskColor() maps wildfire probability values to colours:

  • Green <31% - Low risk
  • Yellow 31-60% - Moderate risk
  • Orange 61-80% - High risk
  • Red >80% - Severe risk

This allows users to quickly understand wildfire risk directly from the device.

Sprint 7 - WiFi Communication

The ESP-01 WiFi module communicates with the Pico through UART serial communication. Using AT commands, the Pico connects to WiFi and sends HTTP POST requests containing sensor and button data to the server.

The server processes this data and returns the predicted wildfire probability (used by the led).

Version 4 - Web Dashboard

Sprint 8 - Dashboard Development

A web dashboard was created to visualise wildfire data in real time. The interface displays:

  • Temperature
  • Humidity
  • Timestamp
  • Wildfire probability

The dashboard connects to the backend using WebSockets, allowing it to update in realtime whenever new sensor data is received.

The dashboard also makes a GET request to the server to retrieve historical data, which is displayed in charts. The charts were implemented using the Chart.js library, including line charts for temperature and humidity trends and a half-doughnut chart for probability.

Sprint 9 - Wildfire Simulator

I added a simulation tool that allows users to test the model. Users can adjust temperature and humidity sliders, which send data to the /predict endpoint in the server. The predicted wildfire risk is returned and displayed on the simulator, allowing users to see how changes in environmental conditions affect wildfire risk.

The website was made responsive to ensure it works well on different devices, using CSS media queries.

Version 5 - Model Testing and Evaluation

Sprint 10 - Model Testing

(The Major problem I encountered. I spent ages trying to fix it. Turned out it was a simple one line fix) The model initially showed 95.7% accuracy. However, further testing using a confusion matrix revealed a major issue: the model predicted every case as non-wildfire (Class 0). While it correctly classified 24,528 non-wildfire cases, it failed to detect any of the 1,097 wildfire cases (0 true positives). This showed that the accuracy was misleading due to severe class imbalance in the dataset.

Sprint 11 - Model Improvement

To fix this issue, I set:

class_weight = "balanced"

This increased the importance of the minority wildfire class.

After retraining:

  • 756 wildfire cases were correctly detected
  • 7,793 false positives occurred

Although false positives increased, this trade-off was acceptable because false positives are safer than false negatives in wildfire prediction.

Another issue was multicollinearity between temperature and humidity (Independent Variables with high correlation with each other). Since both were important features, I kept them but transformed humidity into a new feature: dryness = 100 - humidity. This makes the relationship clearer - higher dryness corresponds to higher wildfire risk.
