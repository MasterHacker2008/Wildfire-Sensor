Wildfire Detector

This project uses an embedded system to collect real-time environmental data and a computer-based model to estimate wildfire risk. The system allows both live monitoring and “what-if” simulations to explore how changes in temperature and humidity affect fire risk.

Meeting the brief

Basic Requirements

I developed a functional embedded system that collects environmental data related to wildfire risk. It uses analogue inputs (temperature and humidity) and a digital input (a button). The system generates a digital output through an LED when risk is high and automatically sends data to a server. After initial setup, it runs independently, continuously monitoring and calculating fire risk.

The collected data is stored and used in my computer-based model. The system simulates how high temperature, low humidity increase wildfire risk, and I tested it under different environmental conditions.

Advanced Requirements

Using Python, I developed a wildfire risk model (logistic regression model) that is trained on an external dataset and is tested on sensor data collected from the embedded system to predict risk. I created two sensor data with simulated drought conditions. I created two “what-if” scenarios: increased temperature and reduced humidity. In both cases, (as predicted) wildfire risk increased.

An adaptive feedback mechanism triggers alerts automatically when risk exceeds a threshold, ensuring the system responds to changing conditions.

Investigation

During planning, I considered several possible forest-related themes:

  • Drought monitoring system (using soil moisture sensors)
  • Forest biodiversity tracker (simulating species population changes)
  • Carbon storage simulation model
  • Wildfire risk detection system

I chose wildfire risk modelling because it allowed for:
  • Clear environmental variables (temperature, humidity)
  • Strong real-world relevance
  • Both real-time detection and predictive modelling
  • Integration of embedded hardware and Python simulation

Plan & Design

Design Objectives

The original plan was to use a Micro:bit as the microcontroller. However, transmitting sensor values (particularly the button as it needs to send an alert) to the website required using USB serial communication. During testing, I discovered that serial communication would occasionally split and merge data packets, making transmission unreliable.

The image here shows the serial output where the temperature 23 sometimes appears as just as 3. Even with using seperators (/) this issue persits, making it immpossible for accurate predictions to be made.

To solve this, I considered sending data directly to the website over WiFi. However, the Micro:bit did not integrate reliably with the ESP serial WiFi module available to me. Due to these limitations, I decided to switch to a Raspberry Pi Pico which works with the esp. Since I had prior experience working with the pico (I used it for my JC tech project) this was not a setback.

Project Options and Design Choices

Research into Different Models

I explored several approaches for predicting wildfire risk:

  • ML Models:
    • Logistic Regression - a standard statistical model.
    • Random Forest - a more complex machine learning model.
  • Simple Mathematical Model
Choosing a Model

I decided against a fully algorithmic approach because it would have required a lot of research and calculations to determine the influence of each factor on wildfire risk. Instead, I opted for a model that could be trained using an existing dataset.

Since my knowledge of machine learning is limited (I initially thought the Random Forest model had something to do with forests), I avoided using complex models and chose the simplest model (the Logistic Regression model) that could still provide reasonable predictions for this scenario.

This allowed me to focus on building the server and programming the Pico (the areas I am strongest in) rather than worrying about implementing advanced machine learning models.

The wildfire dataset contained multiple environmental variables. To simplify the model and improve efficiency, I calculated feature importance using Python to identify the factors most strongly associated with wildfire occurrence.

Ideally, the model would include temperature, precipitation, and solar radiation, as these had the highest importance. However, the sensors for precipitation and solar radiation were not available. Therefore, I selected temperature and humidity as the key variables to train the model, as they were both available and significantly influential.

Create

At first, I planned to run the trained ML model directly on the Raspberry Pi Pico. However, I realized that .joblib models depend on scikit-learn, which isn’t supported on microcontrollers.

Why It Happened:

Microcontrollers like the Pico have significant limitations:

  • Very limited RAM
  • No full Python environment
  • No support for libraries like scikit-learn

Because of these constraints, running a standard trained model directly on the device wasn't feasible.

Resolution:

Instead, I deployed the model on a server, which required a Python-based backend. I had mainly worked with Node.js (A javascript framework to make servers) before, but since the machine learning model was built in Python, using a Python framework made more sense. After some research, I chose FastAPI. Although I was new to it, the clear documentation allowed me tobuild the server by just copy pasting the necessary code with a few tweaks. Now will all the system architecture in place I began programming.

System Architecture

The system collects environmental data (temperature and humidity), sends it to a server, predicts wildfire risk using a trained machine learning model, and stores historical records for analysis. My project follows a client-server architecture with three main components:

  • Microcontroller (Raspberry Pi Pico)
  • Backend Server (FastAPI)
  • The Web App (Frontend)

Raspberry Pi Pico (Microcontroller)

This program runs on a Raspberry Pi Pico and integrates a DHT11 sensor, RGB LED, and ESP-01 module to collect environmental data, send it to a server, and display wildfire risk visually.

1. RGB LED Control (PWM)

The code uses PWM (Pulse Width Modulation) to control the RGB LED. Pins GP16, GP17, and GP18 are configured as PWM outputs for red, green, and blue channels. The setColor() function scales standard 8-bit RGB values (0-255) to the 16-bit PWM range (0-65535). The riskColor() function maps wildfire risk probabilities to color codes: green, yellow, orange, or red, providing a visual risk indicator.

2. ESP-01 Serial Communication (UART)

The ESP-01 module connects via UART (TX/RX pins) and is used for WiFi communication. The send_at() function sends AT commands and handles basic responses. The code initializes the ESP-01, resets it, joins a WiFi network, and sets up TCP/IP connections.

3. HTTP POST via Serial

The program constructs a raw HTTP POST request in JSON format containing temperature, humidity, and an optional manual alert (m_alert). It sends this request over the ESP-01 connection to a server and reads the response to extract the predicted wildfire risk.

4. Sensor & Manual Alert Handling

The DHT11 sensor is periodically read for temperature and humidity. A physical button allows manual alerts, which override normal readings and flash the LED red.

5. Main Loop Logic

The main loop coordinates reading sensor data, preparing the JSON payload, sending it to the server, parsing the server response for wildfire risk, and updating the RGB LED. It also includes error handling.

Backend Server (FastAPI)

This project includes a FastAPI server that collects sensor data, stores it in a Google Sheet, runs wildfire risk predictions, and sends updates to connected clients in real time.

1. Data Handling and Model Prediction

Sensor readings (temperature and humidity) are sent to the server via HTTP POST requests. The server uses a pre-trained logistic regression model (model.pkl) to calculate the probability of wildfire risk (0-100%). The predict() function formats the incoming sensor data for the model and returns the risk probability. Each new record includes a timestamp, sensor readings, and predicted probability.

2. Storage Integration

Instead of a local database, the server stores sensor records in a Google Sheet using the Sheety API. Every new POSTed record is sent to Sheety, ensuring a cloud-based history that can be easily inspected or visualized.

3. Real-Time Updates via WebSockets

The server exposes a WebSocket endpoint (/ws) to push new data to clients immediately when it arrives. An asyncio Event (update_event) signals connected clients when new sensor data is posted, allowing websites or applications to receive live updates without polling the server.

4. API Endpoints

  • GET /data — Fetches all stored sensor records from the Google Sheet as JSON.
  • POST /data — Accepts new sensor data, runs a prediction, stores it in Google Sheets, and notifies WebSocket clients.
  • POST /predict — Returns a risk prediction for given sensor values without storing them.
  • WebSocket /ws — Pushes the latest data to all connected clients whenever new data arrives.

5. Additional Features (Later additions that I made)

  • Supports manual alerts (m_alert) that trigger WebSocket notifications without storing in the database.
  • Enables CORS (Cross-Origin Resource Sharing) so browser clients can interact with the server from different origins. There was a issue where the pico couldn't connect to the server due to CORS restrictions.

Summary: This server is the core backend of the wildfire risk system. It handles sensor data input, risk prediction using a machine learning model, cloud storage, and real-time communication with client applications.

The Web App (Frontend)

Dashboard   data.js

This JavaScript code powers the web dashboard for monitoring wildfire risk in real time.

1. Fetching and Displaying Sensor Data

The code retrieves historical and latest sensor readings from the FastAPI server using HTTP GET requests. Dashboard elements are updated with current temperature, humidity, timestamp, and predicted wildfire probability.

2. Real-Time Updates via WebSocket

A WebSocket connection listens for new data from the server. When new sensor readings arrive, the dashboard automatically updates. If a m_alert (manual alert) is triggered, a popup alert notifies the user immediately.

3. Data Visualization

The dashboard uses Chart.js to display:

  • Line charts for temperature and humidity trends over time.
  • A half-doughnut chart showing the current wildfire risk probability.

Charts dynamically reflect the latest sensor data and provide a visual summary of environmental conditions and risk levels.

Summary: This JavaScript integrates real-time data, alert notifications, and visualizations to provide a live, user-friendly interface for monitoring wildfire conditions.

Simulator   simulator.js

This JavaScript powers the interactive wildfire risk simulator on the website. It allows users to test different temperature and humidity values and see predicted wildfire risk in real time.

1. Input Handling

Monitors changes to the temperature and humidity sliders. The slider’s visual fill dynamically updates to reflect the current value, giving immediate visual feedback.

2. Server Communication

Sends user-selected temperature and humidity values to the FastAPI server’s /predict endpoint using HTTP POST requests. Receives the predicted wildfire risk probability in response.

3. Result Display

Updates the simulator interface to show the predicted risk percentage immediately after submission, providing users with real-time feedback on how environmental conditions affect wildfire risk.

Summary: The simulator provides an interactive tool for exploring wildfire risk predictions and demonstrates the impact of temperature and humidity changes on the probability calculated by the model.

Testing And Evaluation

Problem

The initial logistic regression model achieved a high accuracy of 95.7%, but the confusion matrix revealed a major flaw: The model predicted every case as non-wildfire (Class 0). While it correctly classified 24,528 non-wildfire cases, it failed to detect any of the 1,097 wildfire cases (0 true positives). This demonstrates that accuracy was misleading due to the severe class imbalance. The model was essentially useless for predicting wildfires, as it could not identify any actual wildfire instances.

Solution

This was a huge problem since I didn't know much about machine learning, I had to spend ages trying to figure out how to fix it. Turned out it was a simple one line fix class_weight='balanced' that I had to add to the logistic regression model. This automatically adjusts the weights, giving more importance to the minority class (wildfires occuring). After adding this, the model was able to correctly identify wildfire cases, improving its practical usefulness significantly. The model now correctly detected 756 wildfire cases, although it misclassified 341 and produced 7,793 false positives. Since it is better to have false positives than false negatives in this context, I was happy with this result.
